## 二、DeepSeek 缓存机制（技术基础）

| 参数 | 值 |
|------|-----|
| 最小缓存单元 | **64 tokens** |
| 缓存折扣 | 命中价 = 未命中价 × **1/50 ~ 1/100** |
| 缓存粒度 | 按 API Key 隔离 |
| TTL | 几小时到几天（不活跃自动清空） |
| 支持模型 | V4-Pro ✅ / V4-Flash ✅ |
| 命中标记 | `usage.prompt_cache_hit_tokens` |
| 未命中标记 | `usage.prompt_cache_miss_tokens` |

**缓存原理：**
DeepSeek 对请求的前缀（从头开始的连续 tokens）进行缓存。如果新请求的前缀与之前缓存的前缀匹配，匹配的 tokens 按缓存命中价格计费。

**关键结论：前缀越稳定，缓存越值钱。**

## 三、代理层优化策略

### 策略 1：请求前缀优化（P0）

**问题：** 开发者经常在 system prompt 里放动态内容（时间戳、用户ID、随机数），导致每次请求前缀不同，缓存完全失效。

**代理层做什么：**
```
开发者原始请求 → 代理层自动优化 → 发给 DeepSeek
```

具体动作：
1. **提取动态内容**：从 system prompt 中识别并移除时间戳、随机ID、UUID 等
2. **标准化空白**：多余空格/换行统一为单空格
3. **JSON 标准化**：`sort_keys=True`，确保相同内容产生相同字符串
4. **静态前缀最大化**：确保 system prompt 在最前面，且内容稳定

**效果预估：**
- 优化前：system prompt 含时间戳 → 每次前缀不同 → 缓存命中率 ~0%
- 优化后：system prompt 稳定 → 缓存命中率可达 70-90%
- 省钱：输入 token 费用降低 **50-100 倍**

**实现代码框架：**
```python
import hashlib
import json
import re
from datetime import datetime

def optimize_request(messages: list) -> list:
    """优化请求以最大化缓存命中率"""
    
    optimized = []
    for msg in messages:
        content = msg.get("content", "")
        if msg["role"] == "system":
            # 清洗 system prompt：移除动态内容
            content = clean_system_prompt(content)
            # 标准化空白
            content = normalize_whitespace(content)
        
        optimized.append({
            "role": msg["role"],
            "content": content
        })
    
    return optimized

def clean_system_prompt(text: str) -> str:
    """移除 system prompt 中的动态内容"""
    # 移除各种时间格式
    text = re.sub(r'\d{4}[-/]\d{2}[-/]\d{2}', '[DATE]', text)
    text = re.sub(r'\d{2}:\d{2}(:\d{2})?', '[TIME]', text)
    # 移除 UUID
    text = re.sub(r'[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}', '[UUID]', text)
    # 移除 Unix 时间戳
    text = re.sub(r'\b1[6-9]\d{8}\b', '[TIMESTAMP]', text)
    return text

def normalize_whitespace(text: str) -> str:
    """标准化空白字符"""
    text = re.sub(r'[ \t]+', ' ', text)
    text = re.sub(r'\n{3,}', '\n\n', text)
    return text.strip()
```

### 策略 2：本地响应缓存（P0）

**原理：** 完全相同的请求不需要重复调用上游 API，直接返回缓存结果。

**实现：**
```python
import hashlib
import sqlite3
import time

class ResponseCache:
    """本地响应缓存"""
    
    def __init__(self, db_path="./data/response_cache.db"):
        self.db = sqlite3.connect(db_path)
        self._init_table()
    
    def _init_table(self):
        self.db.execute("""
            CREATE TABLE IF NOT EXISTS response_cache (
                request_hash TEXT PRIMARY KEY,
                response_json TEXT,
                model TEXT,
                created_at REAL,
                ttl_seconds INTEGER DEFAULT 86400,
                hit_count INTEGER DEFAULT 0
            )
        """)
        self.db.commit()
    
    def get(self, model: str, messages: list, temperature: float = 0) -> dict:
        """查找缓存，命中返回响应，未命中返回 None"""
        key = self._make_key(model, messages, temperature)
        row = self.db.execute(
            "SELECT response_json, created_at, ttl_seconds FROM response_cache WHERE request_hash = ?",
            (key,)
        ).fetchone()
        
        if row:
            response_json, created_at, ttl = row
            if time.time() - created_at < ttl:
                self.db.execute(
                    "UPDATE response_cache SET hit_count = hit_count + 1 WHERE request_hash = ?",
                    (key,)
                )
                self.db.commit()
                return json.loads(response_json)
            else:
                # 过期，删除
                self.db.execute("DELETE FROM response_cache WHERE request_hash = ?", (key,))
                self.db.commit()
        
        return None
    
    def put(self, model: str, messages: list, temperature: float, response: dict, ttl: int = 86400):
        """存入缓存"""
        key = self._make_key(model, messages, temperature)
        self.db.execute(
            "INSERT OR REPLACE INTO response_cache (request_hash, response_json, model, created_at, ttl_seconds) VALUES (?, ?, ?, ?, ?)",
            (key, json.dumps(response), model, time.time(), ttl)
        )
        self.db.commit()
    
    def _make_key(self, model: str, messages: list, temperature: float) -> str:
        """生成请求的唯一哈希"""
        content = json.dumps({
            "model": model,
            "messages": messages,
            "temperature": temperature,
        }, sort_keys=True, ensure_ascii=False)
        return hashlib.sha256(content.encode()).hexdigest()
    
    def stats(self) -> dict:
        """缓存统计"""
        total = self.db.execute("SELECT COUNT(*) FROM response_cache").fetchone()[0]
        total_hits = self.db.execute("SELECT SUM(hit_count) FROM response_cache").fetchone()[0] or 0
        return {"cached_entries": total, "total_hits": total_hits}
```

**注意事项：**
- 只对 `temperature=0` 的请求做缓存（确定性输出）
- temperature > 0 的请求每次结果不同，不适合缓存
- 默认 TTL 24 小时，可按 API Key 配置

## 四、整体架构

```
开发者请求
    ↓
[API 代理层]
    ↓
┌─────────────────────────────────────┐
│  1. 本地缓存检查（策略2）             │
│     命中 → 直接返回（省100%）          │
│     未命中 → 继续                     │
├─────────────────────────────────────┤
│  2. 请求前缀优化（策略1）              │
│     - 清洗 system prompt              │
│     - 标准化格式                       │
│     - 记录优化前后的 token 对比         │
├─────────────────────────────────────┤
│  3. 转发到 DeepSeek                   │
│     - 使用用户的 X-Upstream-Key       │
│     - 代理前缀已优化，提高缓存命中率     │
├─────────────────────────────────────┤
│  4. 记录指标（策略3）                   │
│     - DeepSeek 缓存命中/未命中 tokens   │
│     - 本地缓存命中次数                  │
│     - 估算节省金额                      │
├─────────────────────────────────────┤
│  5. 缓存响应（策略2）                   │
│     - temperature=0 的请求存入本地缓存   │
└─────────────────────────────────────┘
    ↓
返回开发者
```

## 七、技术风险

| 风险 | 影响 | 应对 |
|------|------|------|
| DeepSeek 缓存策略变化 | 缓存命中率下降 | 持续监控，及时调整 |
| 本地缓存占用磁盘 | 数据库膨胀 | 设置最大容量 + 自动清理 |
| 请求优化影响效果 | 用户投诉 | 允许用户关闭优化（透传模式） |
| 缓存命中统计不准 | 信任危机 | 直接展示 DeepSeek 原始数据 |
